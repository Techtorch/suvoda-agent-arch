Suvoda AI Workshop — Narrative & Chapter Outline

TechTorch | 3.5-hour workshop | Product engineering managers + IT

The through-line

Every chapter feeds one thesis: software development is shifting from humans writing code with AI help, to humans designing the environments where AI writes code. The room walks in stuck — they've rolled out AI tools and haven't seen the transformation they expected. By the end, they have a new map for why, and a concrete picture of what unlocks the next step change.

What they carry home

  • Formula: Agent = Model + Harness
  • Analogy: Model is the CPU. Harness is the operating system.
  • Principle: Every time an agent fails, engineer a permanent fix so it never fails that way again.
  • Proof: Our internal pipeline — client request in, working prototype out, in hours.

Chapter 1 — Overview of the Macro-Shift in the AI Ecosystem

Goal

Establish the frame. Give the room a coordinate system for where the field is. By end of chapter, they should see that what they're experiencing — AI adopted but not transformative — is a known plateau with a known path through it.

Talking points

  • Software has moved through three paradigms: code written by humans, models trained on data, systems instructed in natural language — all three coexist in modern applications
  • The natural-language layer is absorbing work that used to require hard-coded rules and bespoke models
  • The engineering craft has evolved in parallel: prompt engineering → context engineering → environment engineering
  • LLMs behave less like a utility and more like a new computing substrate — something you build on top of, not something you plug into
  • Real cognitive limits of these systems: no retention across sessions, uneven capability, sensitivity to what you put in their context window
  • Productive design stance: partial autonomy with humans in the loop — augmentation, not replacement
  • Working definition of an agent we'll use all day: a model using tools in a loop to achieve a goal
  • The macro thesis: software development is shifting from humans writing code with AI help, to humans designing the environments where AI writes code
Transition to Chapter 2: "That's the shift at the ecosystem level. What does it actually mean for the way software gets built day-to-day?"

Chapter 2 — Traditional SDLC vs an AI-Native Approach

Goal

Translate the macro-shift into the daily reality of the engineering managers in the room. By end of chapter, they should see their current SDLC as an artifact of a specific era, not a neutral given.

Talking points

  • Traditional SDLC was designed around human-speed production and human-eye review: sequential phases, gated handoffs, batched releases
  • AI-native SDLC is continuous and parallel: design, build, and test happen simultaneously, with agents alongside humans across all three
  • Shift 1 — Requirements become specs the agent executes against. Structured, context-rich specifications replace ambiguous requirements docs
  • Shift 2 — Writing code → curating code. Production becomes cheap; judgment becomes expensive. Engineers shift from authors to intent-definers and validators
  • Shift 3 — Testing moves from a gate to a continuous property. Test generation, regression detection, and compliance checks run continuously. This is the shift that matters most in regulated environments
  • Shift 4 — Documentation becomes the agent's operating context. If it isn't in the repo, it doesn't exist for the agent
  • The pattern that captures all four: continuous design, continuous build, continuous test — the AI-native generalization of CI/CD
  • Team shape: smaller, more senior; architect and validation roles become more important; junior roles shift toward agent orchestration and eval design
  • Measurement: velocity metrics age out. New measures — agent leverage ratio, eval coverage, time from intent to production, regression rate
Transition to Chapter 3: "So that's the workflow shift. The obvious next question — what are we actually working with? And what's the vocabulary we'll use for the rest of the day?"

Chapter 3 — Tools, Tech Stacks & Agentic Development Patterns

Goal

Build shared vocabulary and introduce the harness concept. This is the landscape chapter and the conceptual chapter combined. By end of it, the room has the formula, the mental model, and the component language we'll use for the rest of the day.

Talking points

Definitional grounding

  • Agent = a model using tools in a loop to achieve a goal
  • Three components: environment, tools, system prompt. Model gets invoked in a loop
  • Three principles we apply: don't build an agent when a workflow will do; keep it simple because upfront complexity kills iteration; think from the agent's point of view — it only sees what's in its context window

The tools landscape — three zones

  • Consumer-grade AI coding tools — fit for non-developer prototyping, unfit for regulated enterprise
  • Developer-assistant tools — where most enterprise engineers currently operate
  • Agentic coding platforms — the frontier; tools that run autonomously for minutes to hours
  • Productivity pattern: running multiple coding agents in parallel against different parts of a codebase

The six-layer AI-native tech stack

  • Model layer — routing across a portfolio by task and cost, not picking one
  • Agent layer — orchestration runtime; heavy frameworks slow production
  • Tool and protocol layer — Model Context Protocol (MCP) as the open standard; we design to MCP to preserve portability
  • Memory and knowledge layer — working, short-term, long-term, episodic, semantic, procedural memory; most enterprise AI has only working memory; knowledge graphs for multi-hop auditable retrieval
  • Skills layer — modular, versioned capabilities loaded on demand via progressive disclosure; the shift from prompt engineering to capability engineering
  • Eval and observability layer — how you know any of this is working; the layer most organizations have nothing in

Introducing the harness

  • Formula: Agent = Model + Harness
  • Model reasons; harness does everything else — context, constraints, feedback, validation
  • Analogy: model is the CPU, context window is the RAM, harness is the operating system, agent is the application
  • Operating principle: every time an agent fails, engineer a permanent fix so it never fails that way again. Failures become compound infrastructure
  • Two primary classes of components:
    • Guides — feedforward controls that direct the agent before it acts
    • Sensors — feedback controls that observe and validate what the agent did

Agentic development patterns we use

  • Workflows vs. agents — don't build an agent when a workflow will do
  • Sub-agent composition — narrow specialists coordinated by a planner; monolithic agents can't be debugged in regulated contexts
  • Actor-critic loops — actor produces, evaluator assesses, reflector critiques, next attempt improves; fails silently if the evaluator can't detect the original error
Transition to Chapter 4: "You have the formula, the stack, the vocabulary. What does this actually look like in a real codebase?"

Chapter 4 — TechTorch's Approach for AI-Native Product Development in Practice

Goal

Make the harness concrete. Walk through a real repo structure so every abstract concept from Chapter 3 gets a file to look at. By end of this chapter, the room should be able to picture what a harness physically is in their own codebase.

Talking points

Framing the walkthrough

  • We've built a reference repo — a trivial service wrapped in a complete harness
  • The point isn't the service. The point is what surrounds it
  • This is the pattern we use on engagements, simplified so nothing in the domain competes with the architecture

View 1 — The guides (what the agent reads before it acts)

  • AGENTS.md at the root — system-prompt-equivalent every agent reads at task start
  • .claude/skills/ — modular capabilities loaded on demand via progressive disclosure
  • docs/architecture/invariants.md — properties that must never change; deterministic guarantees the sensor layer enforces
  • docs/architecture/decisions/ — ADRs explaining why the code is shaped the way it is, preventing agents from "fixing" intentional decisions
  • docs/specs/ — structured specifications the agent executes against

View 2 — The sensors (what checks the agent's work)

  • harness/sensors/linters/ — deterministic enforcement of architectural rules; block the PR when standards are violated
  • harness/sensors/drift_detectors/ — the garbage collector pattern; runs nightly, finds inconsistencies, opens issues for agents to fix
  • evals/ — datasets, rubrics, runners; eval scores gate merges
  • tests/ — unit, integration, property-based; property tests assert the invariants declared in guides actually hold
  • .github/workflows/ — CI as the deterministic backstop where guides and sensors come together

View 3 — How it compounds

  • Every sensor enforces something declared in a guide
  • Every linter came from a past failure that got encoded so it never happens again
  • Every eval case came from an edge case someone caught
  • The directory structure is the accumulated learning of the team, made executable
  • A new agent dropped into this repo on day 90 operates in a dramatically richer environment than one dropped in on day 1 — not because the model got better, but because the environment did
  • This is what "every failure becomes infrastructure" looks like in a file tree

Our four operating principles, visible in the walkthrough

  • Build the harness before the agent — sensor layer ships first
  • Treat failures as infrastructure — every mistake becomes a permanent sensor
  • Deterministic constraints over probabilistic compliance — linters, not prompt pleas
  • Humans steer, agents execute — every workflow has explicit autonomy boundaries encoded in the system
Transition to Chapter 5: "That's the pattern and what it looks like in a reference repo. Does it actually work in production, on real business problems, at speed? Let us show you how we use this internally."

Chapter 5 — Case Study: Client Request to Working Prototype in Hours

Goal

Prove the approach works by showing it running inside our own business. TechTorch is an AI consulting firm — the way we respond to client requests is itself an AI-native workflow. By end of this chapter, the room should see that we don't just advise on this. We operate this way, on real work, every day.

Talking points

The problem we were solving inside TechTorch

  • Traditional consulting response: discovery → deck → scoping → proposal → build. Clients wait weeks to see anything tangible
  • We wanted to flip the model. A client describes a problem; we show them a working prototype the same day — something they can click through, react to, and shape
  • The goal: replace decks and slideware with working software as the first artifact of the engagement. Faster to value, cheaper to iterate, more honest about what's actually feasible

The pipeline — two halves of one system

  • Half 1: Intake agent. Structured interview that takes a rough client idea and produces a high-fidelity brief. The agent asks targeted questions, tracks a blueprint-confidence score, and only passes the brief downstream when it has enough to build against
  • Half 2: Build agent. Takes the structured brief and produces a working prototype autonomously. Not a wireframe. Not a mockup. A running application the client can use
  • Round-trip time: hours, not weeks

The harness that made it work — same pattern from Chapter 4

  • Guides: intake question templates, spec schemas for what a complete brief looks like, architectural constraints for the output, skill files for common UI and data patterns
  • Sensors: blueprint-confidence scoring at intake, schema validators on briefs before handoff, output validators checking generated prototypes against the brief, eval rubrics on code and UX quality
  • Autonomy boundaries: intake is human-driven with agent guidance, brief handoff is agent-validated and human-approved, prototype generation is fully autonomous, client delivery is always human-reviewed
  • Compounding: every early failure — a brief that looked complete but wasn't, a prototype that technically met the spec but wasn't usable — became a permanent sensor. The pipeline improves every week because the harness keeps accumulating

Honest lessons

  • Early versions asked too few intake questions and produced briefs that looked good and weren't
  • Early prototype generation produced technically-correct slop; we added UX-quality evals, not just functional correctness checks
  • We built one mega-agent for the whole pipeline first; it was impossible to debug. Rebuilt as sub-agents with clear contracts
  • We over-indexed on the model for months before accepting the environment was the bottleneck

The outcome

  • Client request to working prototype: hours instead of weeks
  • What our team spends time on has changed. Less time producing artifacts, more time on judgment calls — which problems to solve, how to shape the solution, what to push back on
  • The type of engagement has changed too. We start with working software, not slideware. Clients give sharper feedback because they're reacting to something real

What this means for Suvoda

  • Same pattern applies to any domain where a structured brief can produce a structured output
  • For you: protocol amendment impact analysis, IRT configuration generation, regulatory submission drafts, validation documentation
  • The investment isn't picking a model. It's building the harness for your specific domain — guides that encode your regulatory requirements, sensors that check outputs deterministically, evals that capture what "good" means in clinical trials
  • It's feasible, it's working at TechTorch on real work, and it's the same investment profile we'd recommend here
Transition to Chapter 6: "You've seen the frame, the pattern in a repo, and the pattern running at TechTorch. Last piece — what do you actually do on Monday morning?"

Chapter 6 — Good Habits and Pitfalls to Avoid

Goal

Send them home with specific, actionable habits and a concrete method for the next 90 days. By end of it, every person in the room should be able to name one thing they'll do next week to start building their own harness.

Talking points

Seven paired habits and pitfalls

  • Deterministic enforcement over probabilistic compliance. Wire constraints into linters and CI gates. Asking the agent nicely in a prompt does not survive audit
  • Eval-first development over unmeasurable agents. Write the eval set before the agent. If you can't measure it, you can't ship it into a regulated path
  • Narrow composable agents over mega-agents. Small, single-purpose agents with clear contracts. Monolithic agents can't be debugged when they fail
  • Versioned prompts, skills, and evals over prompt drift. Treat them as code. Untracked changes are how working systems quietly become broken ones
  • Deliberate context engineering over context bloat. What goes into the context window is a design choice. More context does not mean better outcomes
  • Explicit autonomy boundaries over quiet agent actions. Define what's unsupervised, what needs approval, what's off-limits; encode the boundaries in the system
  • Continuous validation in the workflow over validation as a gate. Build validation into the harness so it runs continuously; month-long manual validation phases kill the velocity AI gave you

The meta-habit that ties everything together

  • When an agent fails, the instinct is to patch the prompt. The discipline is to engineer a permanent fix into the harness so that failure never happens again
  • Every failure becomes compound infrastructure. This one practice separates teams that plateau from teams that compound

The 90-day method we recommend

  • Pick one workflow in your organization
  • Map its harness across guides, sensors, tools, memory, control flow
  • Identify the single missing guide or sensor that matters most
  • Build that one thing next week
  • The week after, find the next gap
  • Within a quarter you have an operating environment competitors can't replicate by buying tools

Closing thought

  • The real shift isn't AI adoption. It's the shift from optimizing the model to engineering the environment. That's the gap between teams that plateau and teams that compound
